### Obsidian Task Monitor 核心设计文档

**1. 系统定位**
一个轻量级的 Python 后台守护进程 (Daemon), 通过桥接 Obsidian 每日计划与 TaskNote 执行数据, 实现自动化的任务防沉迷与动态分流指引 (强制中断 + 建议流向).

**2. 数据与架构**

- **配置源 (输入 A):** `DailyNotes/YYYY-MM-DD.md`. 提供当天的任务名称与番茄钟配额.
- **状态源 (输入 B):** `TaskNotes/data.json`. 提供客观的番茄钟执行历史.
- **输出端:** 基于 `tkinter` 的自定义富文本置顶弹窗 (支持红绿双色展示与系统提示音).

**3. 核心执行逻辑**

- **低频触发:** 每 3 秒轮询一次输入 A 和 B 的文件修改时间 (`mtime`).
- **精准配额解析:** 正则提取当日笔记中的 Obsidian 双链任务配额.
  - 目标格式: `- [[任务文件名]] : 数字` (例如: `- [[new ankify pass - ddca.solid]] : 5`).
  - 正则表达式: `^[-*]\s+\[\[(.*?)\]\]\s*:\s*(\d+)`. Group 1 精确提取纯文件名, Group 2 提取限额数字.
- **严谨进度统计:** 遍历 JSON 历史, 筛选出 `startTime` 为今日, `type` 为 `work`, 且 `completed` 为 `true` 的条目.
  - 匹配规则: 提取双链中的文件名后, 使用严格的路径后缀匹配 (例如: `task_path.lower().endswith(f"{task_name.lower()}.md")`) 来确认该番茄钟归属, 彻底消除模糊匹配造成的误判.
- **状态动态归类:** 每次检查时, 将计划清单中的任务分为两组:
  - **超载组 (Overloaded):** `已完成数 >= 配额`.
  - **待办组 (Pending):** `已完成数 < 配额`.
- **精准狙击打断:** 当某个任务处于超载组, 且其当前 `已完成数` 大于上次触发警告时的记录值, 立即触发中断弹窗.

**4. 报警 UI 设计**
彻底弃用系统默认的单调对话框, 改用 `tkinter.Toplevel` 绘制包含"推拉"双向反馈的自定义窗口:

- **红色警告区 (Stop):** 居中醒目显示当前正在沉迷的超额任务. 使用大号红色字体 (例如: "🛑 你在 [new ankify pass - ddca.solid] 上已花费 6/5 个番茄钟, 严重超时!").
- **绿色引导区 (Go):** 在下方动态列出"待办组"中尚有盈余的任务与当前进度. 使用绿色字体 (例如: "🟢 请立即切换至: [CS149 Kayvon part] (已做 1/4)").

**5. 状态管理与容错 (Edge Cases)**

- **阶梯式防骚扰机制:** 内存中维护一个字典 `last_alerted_count`, 记录每个任务上次触发警告时的完成数. 如果用户无视警告继续"头铁", 每完成一个该任务的新番茄钟, 计数器增加并会再次触发弹窗. 做到"只要继续超额, 每多完成一个就警告一次".
- **无缝跨夜:** 每次轮询时校验当前系统时间. 一旦跨越午夜零点, 自动刷新目标文件路径为次日笔记, 并清空 `last_alerted_count` 字典.
- **静默重试:** 针对文件尚未创建, 或者与 Obsidian 插件发生极小概率的 JSON 并发读写冲突, 均通过 `try-except` 捕获异常并静默 `pass`, 等待下一个 3 秒周期重试, 保证后台进程绝对稳定.
